## 2025-02-18 - String Slicing and OS Calls vs Pathlib Overheads
**Learning:** In loops over files (e.g. grouping UI lists), instantiating `pathlib.Path` objects just to call `.stem` or `p.stat().st_mtime` adds massive cumulative performance overhead compared to basic string manipulation (`name.rsplit(".", 1)[0]`) and `os.stat`. Additionally, wrapping a function in `@functools.lru_cache` provides zero benefit if the input parameter is always unique across the list iteration (e.g., unique full file names passed to `get_name_prefix`), merely adding cache management overhead.
**Action:** When working with thousands of files inside loops, bypass `pathlib` for lightweight extraction and stick to `os` and `str` methods. Always ensure that functions marked with `lru_cache` actually receive overlapping input values before adding the decorator.

## 2025-02-18 - Replacing rglob with os.walk for Directory Exclusion
**Learning:** Using `Path.rglob("*")` followed by a post-filter to check for excluded directories causes the program to unnecessarily traverse deep into ignored subtrees. While replacing this with `os.walk` and modifying the `dirs[:]` list in place provides massive speedups, it is crucial to carefully preserve any dynamic behavior (like mocked feature toggles in tests that bypass the exclusion check). Hardcoding the exclusion logic without wrapping it in the original toggle functions can lead to functional regressions and failing tests.
**Action:** When migrating from `rglob` to `os.walk` for performance, always ensure that pre-existing functions or flags used to bypass or control exclusions are correctly integrated into the new pruning logic, keeping the behavior functionally identical while benefiting from the O(1) directory skip.
